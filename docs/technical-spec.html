SignSafe Technical Specification

Transaction analysis and interception architecture for a Chrome Solana security extension

SignSafe is a Chrome Manifest V3 extension that intercepts Solana wallet signing requests in the page context, simulates transaction effects through Solana RPC, converts the result into a structured risk model, sends those facts to OpenAI for user-facing explanation, and requires explicit user approval before the original wallet method is allowed to continue.

Runtime Model Chrome MV3

Content script at document_start, page-world hook injection, service worker analysis, iframe overlay UI.

Current Providers OpenAI + Solana RPC

Solana simulation happens first. OpenAI receives parsed facts and heuristic reason codes, not a raw wallet popup summary.

Wallet methods: signTransaction / signAllTransactions / sendTransaction / signAndSendTransaction / signMessage Network target: Devnet by default Local test harness: localhost devnet dApp

Primary User Flows

1. Intercept

Wallet method capture

The page hook wraps Solana provider methods and some request(...) variants before the dApp reaches the real wallet API.

2. Analyze

Simulation and heuristics

Transactions are base64-serialized, simulated against Solana RPC, then normalized into facts like SOL delta, token delta, and invoked programs.

3. Explain

Model-assisted summary

OpenAI receives structured facts plus risk codes and returns plain-English summary, actions, risk reasons, and verdict text.

4. Decide

Overlay gate

The wallet method resumes only after the user approves inside the SignSafe overlay. Otherwise the request is blocked.

Transaction Workflow

Transaction methods

  1. The dApp invokes a transaction-related wallet method.
  2. src/page/page-hook.js intercepts and posts an ANALYZE_REQUEST.
  3. src/content/content-script.js creates an overlay session and shows a loading state.
  4. The content script sends ANALYZE_TX to the MV3 background worker.
  5. shared/background-analysis.js simulates, parses, evaluates heuristics, and optionally queries OpenAI.
  6. The content script renders the verdict and waits for a decision.
  7. If approved, the original wallet method executes in the page hook.

signMessage workflow

  1. The page hook intercepts signMessage or a matching request path.
  2. There is no on-chain simulation because the payload is not a transaction.
  3. The content script opens a heuristic-only warning overlay with a readable message preview when possible.
  4. The user can block or allow the blind-signature request.
Current behavior: raw message review is conservative by default and treated as a review-risk action, not safe.

Current Analysis Pipeline

1. Solana simulation

Uses JSON-RPC simulateTransaction with base64 transaction input and replacement of recent blockhash for analysis reliability.

2. Deterministic checks

Heuristics inspect total SOL outflow, token drain patterns, multi-transfer batches, unknown programs, simulation failure, and raw-message signatures.

3. OpenAI explanation

OpenAI receives normalized facts and reason codes, then generates user-facing explanation text while the deterministic risk floor remains authoritative.

Important: the current product is not asking AI to infer everything from raw transaction bytes. It first uses Solana simulation and structured heuristics, then uses AI as an explanation layer.

Key Components

Path Role Responsibility
src/page/page-hook.js Page-world interception Wraps wallet provider methods, serializes requests, and gates real wallet execution behind approval.
src/content/content-script.js Bridge orchestration Receives page bridge messages, coordinates background analysis, and controls the approval lifecycle.
src/content/overlay-session.js Overlay session helper Creates the iframe overlay, posts render messages, and resolves user decisions.
src/background/background.js MV3 entrypoint Receives runtime messages and delegates analysis to the shared analysis service.
shared/background-analysis.js Analysis engine Performs Solana simulation, parsing, heuristics, OpenAI explanation, fallback handling, and caching.
src/overlay/overlay.js Overlay runtime Renders verdict UI, risk labels, facts, batch summaries, and decision buttons.
shared/constants.js Protocol contract Defines channels, message types, and shared storage keys across all runtime surfaces.

Risk Semantics

Risk Meaning Examples
Safe Simulation and heuristics show behavior consistent with the apparent intent. Simple swap-like or low-risk balance change with known programs.
Review Behavior may be legitimate but needs inspection or lacks strong certainty. Unknown program interaction, large SOL outflow, raw message signing, or complex mint/setup behavior.
Danger Deterministic checks indicate a strong risk pattern. Simulation failure, token drain pattern, or suspicious asset movement without clear offsetting inflow.

Configuration and Local Development

Extension setup

  • Load unpacked from the repository root.
  • Set an OpenAI API key in the options page or use local-only fallback from .env.local.
  • Current default RPC endpoint is Solana devnet.

Local harness

Serve the local devnet dApp for deterministic testing:

python3 -m http.server 8788 --directory test-dapp

Then test Connect, Sign Message, Sign Transaction, and Send Transaction.

Known Constraints

Provider interception

Coverage is best-effort across direct wallet methods and some request-style paths. Wallet-standard edge cases can still vary by dApp.

Client-side key usage

The MVP uses a client-side OpenAI key for local testing. This is acceptable for demos, not for a production release.

Simulation limits

Versioning, RPC anomalies, and simulation failure still degrade to conservative review behavior rather than guaranteed interpretation.
